# Shared Shadcn Design System Architecture

## Goal

Create a shadcn-first shared design system package that both the Electron desktop renderer and TanStack/Vite web app use for visual primitives, app-level UI patterns, and design tokens. The package should reduce custom CSS, make visual decisions consistent across apps, and keep app-specific behavior in the apps.

User decisions:

- Use a Tailwind/shadcn-first architecture.
- Focus web rollout on the TanStack/Vite React app, not `apps/web/src/server.ts` HTML-string rendering.
- Optimize the first implementation for a broad migration, not just scaffolding.

## Recommended Architecture

Add a new workspace package:

```text
packages/ui/
  package.json
  tsconfig.json
  src/
    index.ts
    lib/utils.ts
    styles/globals.css
    components/ui/button.tsx
    components/ui/badge.tsx
    components/ui/card.tsx
    components/ui/input.tsx
    components/ui/label.tsx
    components/ui/separator.tsx
    components/ui/tooltip.tsx
    components/app/status-badge.tsx
    components/app/section-card.tsx
    components/app/permission-card.tsx
    components/app/floating-suggestion-bar.tsx
```

Package responsibilities:

- Own shared Tailwind/shadcn tokens in `packages/ui/src/styles/globals.css`.
- Own shadcn primitives under `components/ui`.
- Own reusable Tabb app patterns under `components/app`.
- Export a stable public API from `packages/ui/src/index.ts`.
- Stay browser-safe: no Electron, Bun server, filesystem, IPC, TanStack Router, or auth dependencies.
- Accept behavior through props and callbacks.

App responsibilities:

- Desktop keeps Electron window state, IPC calls, permission flows, and overlay mechanics.
- Web keeps routing, auth, billing flow logic, and page composition.
- Both apps import shared CSS once and import components from `@tabb/ui`.

## Implementation Plan

### 1. Add Tailwind/Shadcn Foundation

- Add Tailwind and required tooling using Bun.
- Add `tailwind.config.ts` at the repo root.
- Include these content globs:

```ts
"./apps/web/src/**/*.{ts,tsx}",
"./apps/desktop/src/renderer/src/**/*.{ts,tsx}",
"./packages/ui/src/**/*.{ts,tsx}"
```

- Add `components.json` configured so shadcn output is canonicalized into `packages/ui`.
- Add `@tabb/ui` to root `tsconfig.json` paths.
- Update root `tsconfig.json` include to cover `packages/**/*.tsx`.
- Add `@tabb/ui` as a workspace dependency of `apps/web` and `apps/desktop`.

Prefer root-level Tailwind config over per-app configs so token changes apply everywhere.

### 2. Create `@tabb/ui`

- Add `packages/ui/package.json` matching the repo‚Äôs simple package style, with explicit exports:

```json
{
  "name": "@tabb/ui",
  "type": "module",
  "private": true,
  "main": "src/index.ts",
  "exports": {
    ".": "./src/index.ts",
    "./styles.css": "./src/styles/globals.css"
  }
}
```

- Put React in `peerDependencies`.
- Put shadcn runtime dependencies in `dependencies`: `class-variance-authority`, `clsx`, `tailwind-merge`, `lucide-react`, and Radix packages as components require them.
- Add `packages/ui/src/lib/utils.ts` with the standard `cn(...inputs)` helper.
- Add `packages/ui/src/styles/globals.css` with Tailwind imports and shadcn CSS variables.

Token direction:

- Preserve the desktop app‚Äôs dark/glass identity as the default `.dark` or desktop surface theme.
- Preserve the web app‚Äôs warm marketing feel through the same semantic tokens, not duplicated hardcoded CSS.
- Use shadcn semantic variables: `--background`, `--foreground`, `--card`, `--card-foreground`, `--primary`, `--primary-foreground`, `--secondary`, `--muted`, `--muted-foreground`, `--accent`, `--border`, `--input`, `--ring`, `--radius`.
- Add Tabb-specific tokens only when shadcn tokens are insufficient, such as overlay/glass/easing tokens.

### 3. Add Shadcn Primitives

Start with primitives needed by both apps and current CSS classes:

- `Button`
- `Badge`
- `Card`
- `Input`
- `Label`
- `Separator`
- `Tooltip` if needed by overlay/onboarding

Use CVA for component variants rather than custom global CSS classes.

Required variants:

- Button: `default`, `secondary`, `ghost`, possibly `link`.
- Badge: `default`, `secondary`, `outline`, plus app-level status mapping in `StatusBadge`.
- Card: standard shadcn `Card`, `CardHeader`, `CardTitle`, `CardDescription`, `CardContent`, `CardFooter`.

Export primitives from `packages/ui/src/index.ts`.

### 4. Migrate Shared App Components

Move presentational desktop components into `packages/ui/src/components/app` and rewrite them with shadcn primitives/Tailwind classes:

- `Button.tsx` becomes usage of shared `Button`; delete local component after import migration.
- `StatusBadge.tsx` becomes `StatusBadge` in `@tabb/ui`, implemented with `Badge`.
- `SectionCard.tsx` becomes `SectionCard` in `@tabb/ui`, implemented with `Card` and glass variants/classes.
- `PermissionCard.tsx` becomes `PermissionCard` in `@tabb/ui`; keep same props where possible.
- `FloatingSuggestionBar.tsx` becomes `FloatingSuggestionBar` in `@tabb/ui`; keep `Suggestion` type exported; keep `onAccept` callback prop.

Do not move `DebugContextCard` initially unless it is used outside desktop. It can stay desktop-specific until a second pass because it is debug/overlay-specific and has no web reuse.

### 5. Integrate Desktop Renderer

- Replace CSS imports in `apps/desktop/src/renderer/src/main.tsx` with `@tabb/ui/styles.css` plus any temporary desktop-only CSS still needed.
- Replace imports in `OnboardingSurface.tsx` and `OverlaySurface.tsx` with `@tabb/ui` imports.
- Convert screen layout classes to Tailwind utility classes as part of the broad migration.
- Keep Electron-only CSS/mechanics temporarily where needed:
  - `-webkit-app-region`
  - transparent overlay root behavior
  - pointer-event behavior for overlay windows
  - body `data-surface` backgrounds until equivalent theme/surface classes are proven
- Delete or shrink desktop CSS files only after their classes have no remaining references.

### 6. Integrate Web Routes

- Import `@tabb/ui/styles.css` from `apps/web/src/routes/__root.tsx`.
- Replace `.button`, `.card`, `.panel`, `.muted`, `.lead`, `.hero`, `.grid`, `.pricing-grid`, and `.dashboard-grid` usages with shared components and Tailwind classes.
- Replace inline styles in `index.tsx`, `login.tsx`, and `signup.tsx` with Tailwind utility classes.
- Keep page-specific marketing composition in routes, but no page should define reusable button/card/input styling.
- Reduce `apps/web/src/styles.css` to app-only leftovers or delete it if fully replaced.

### 7. Cleanup Duplicate Styling And Dependencies

- Remove local desktop UI components after all imports move to `@tabb/ui`.
- Remove dead CSS rules from desktop and web.
- Move shared UI dependencies out of apps when no longer directly imported by app code.
- Leave `apps/web/src/server.ts` alone unless it causes build/typecheck issues; it is outside the selected design system rollout path.

## Migration Rules

- New reusable UI belongs in `packages/ui`, not app-local CSS.
- App-local CSS is allowed only for platform mechanics or truly page-specific one-offs.
- Prefer shadcn component variants before custom classes.
- Prefer Tailwind utilities over new CSS selectors.
- Prefer semantic CSS variables over hardcoded colors.
- Do not introduce Electron, TanStack Router, auth, billing, or Bun server dependencies into `@tabb/ui`.
- Keep public exports intentional; avoid exposing every internal file by default.

## Risks And Mitigations

- Tailwind may miss classes from `packages/ui`: include `packages/ui/src/**/*.{ts,tsx}` in content globs.
- CSS package export may not resolve in both Vite pipelines: first try `import "@tabb/ui/styles.css"`; if needed, adjust `exports` before using relative imports.
- Duplicate React copies can break hooks: make React/React DOM peer dependencies of `@tabb/ui`.
- Desktop overlay CSS has platform behavior mixed with visual styling: migrate visuals first and keep overlay mechanics until manually verified.
- Broad migration can become too large: migrate by component family, running builds after each family.
- shadcn CLI may prefer app-local output: generate into `packages/ui` manually or move generated files immediately so `packages/ui` remains canonical.

## Verification

Run after dependency/config work and after each migration batch:

```sh
bun install
bun run typecheck
bun run --cwd apps/web build
bun run --cwd apps/desktop build
```

Manual checks:

- Web home, pricing, download, login, signup, dashboard, and logout routes render with shared styles.
- Desktop onboarding renders correctly and permission buttons still call the existing desktop callbacks.
- Desktop overlay still appears transparent, centered, clickable only where intended, and accepts suggestions.
- Shared `Button`, `Badge`, and `Card` look consistent across web and desktop while respecting light/dark/surface tokens.
- No references remain to deleted local desktop components or removed CSS classes.
