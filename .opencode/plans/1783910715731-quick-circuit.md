# Reliable OpenCode Context for Local Suggestions

## Finding

OpenCode context is wired into local inference on the happy path: `NativeSuggestionSession` attaches `snapshot.appContext`, `local-inference-prototype.ts` passes it to `createSuggestionMessages`, and the shared suggestion policy includes it as background in the llama.cpp request.

It is not reliable yet:

- The SQLite lookup is fire-and-forget, so the first 100 ms-debounced local request can run before conversation context is cached.
- Cache completion does not notify or invalidate the suggestion loop.
- App Context is absent from `contextHash`, allowing requests and visible suggestions generated from stale context to survive terminal/session changes.
- `observe()` drops newer valid observations while a query is in flight.
- Cache identity is only `windowId:title`; default-title OpenCode sessions can reuse another session's conversation in the same window.
- Existing tests prove the pieces independently, but not local DB -> App Context -> local llama prompt behavior or timing races.

## Recommended Changes

### 1. Make App Context observable and revisioned

Update `apps/desktop/src/main/app-context-extractor.ts` and `apps/desktop/src/main/opencode-session-context.ts`:

- Expose App Context state as `{ snapshot, revision, pending }` rather than an invisible mutable cache.
- Increment `revision` whenever the effective normalized conversation candidate changes, clears, or becomes invalid for the active terminal.
- Add a subscription callback so cache publication and invalidation notify the native suggestion session.
- Keep raw terminal/database text in memory only; revisions and identity fingerprints must be hashes, not raw content.

### 2. Make OpenCode observation latest-wins

Refactor `apps/desktop/src/main/opencode-session-context.ts`:

- Record the latest desired terminal observation even while SQLite is in flight.
- Ignore obsolete query results and immediately process the newest queued observation after the current subprocess settles.
- Invalidate the previous candidate while a materially different target is being resolved; never serve the old candidate during refresh.
- Separate stable target identity from changing terminal text.
- For unique non-default titles, retain the binding only while title/window evidence remains compatible.
- For generic `OpenCode` titles, retain a matched session only while transcript excerpts continue to corroborate it; otherwise clear and rematch.
- Do not use recency as an identity signal and continue to abstain on ambiguous matches.

### 3. Include effective App Context in suggestion identity

Update `apps/desktop/src/main/native-suggestion-session.ts`:

- Derive a final request identity from the base Typing Context hash plus the App Context revision/content fingerprint.
- Use that identity for unchanged-context suppression, local request cancellation, partial-result validation, visible-suggestion validity, and diagnostics.
- On App Context subscription changes, invoke the same context-change path used by typing edits so stale local work is aborted and a fresh request is debounced.
- Preserve Ghostty keystrokes as authoritative unfinished text; terminal and OpenCode conversation remain suggestion-only App Context.

### 4. Add a bounded readiness hold for first local requests

Update `apps/desktop/src/main/native-suggestion-session.ts` and, only if needed for a clean seam, `apps/desktop/src/main/suggestion-loop.ts`:

- When OpenCode is positively detected and its conversation lookup is pending, hold the automatic local request for a short bounded grace period (target 150-200 ms).
- If context becomes ready during the hold, request once with the final context revision.
- If the grace period expires, allow terminal-only fallback rather than blocking autocomplete for the SQLite 500 ms maximum.
- If conversation context arrives after fallback inference starts, its revision change must abort that request and debounce a replacement.
- Explicit cloud requests should use the newest ready cache immediately and must not wait on a long database query.

### 5. Preserve useful prompt context deliberately

Review `packages/suggestion-policy/src/index.ts` and context candidate ordering:

- Keep OpenCode conversation before generic terminal fallback, but reserve a small terminal-context budget so a 600-character conversation cannot evict the active visible prompt entirely.
- Continue treating both fragments as background and the reconstructed terminal input as the only unfinished text.
- Preserve newest-tail semantics and whole-word boundaries.

### 6. Add end-to-end and race regression tests

Update/add tests in:

- `tests/desktop-opencode-context.test.ts`
- `tests/desktop-native-loop.test.ts`
- `tests/desktop-local-inference-prototype.test.ts`
- `tests/desktop-app-context.test.ts`

Cover:

- Matched SQLite conversation reaches the actual local llama request messages.
- Conversation and terminal fragments have the intended order and reserved prompt budgets.
- Query completion before the readiness deadline produces one context-rich request.
- Query completion after fallback starts aborts stale work and replaces it.
- App Context-only changes alter request identity and invalidate partial/visible suggestions.
- A newer terminal observation arriving during an in-flight query wins.
- Same-window/default-title session switches never reuse the prior conversation.
- Clear/pause/app-switch during a query prevents stale publication.
- Query timeout/failure falls back to terminal-only context without deadlock.
- App Context remains memory-ineligible and identifiers/directories remain absent from request payloads.
