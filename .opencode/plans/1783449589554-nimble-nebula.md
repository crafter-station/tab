# Build Web App With TanStack Start

## Goal

Replace the current string-rendered `apps/web` surface with a TanStack Start React app that includes:

- Public landing page and pricing/download surfaces.
- Auth flows in scope: sign in, sign up, sign out, protected routes, and existing desktop device handoff.
- Authenticated dashboard/account area for account configuration, usage, billing, devices/permissions, and Personal Memory controls.
- Compatibility with the existing API app as the source of truth for Better Auth, billing, quota, devices, and memory.

## Current State

- `apps/web/src/server.ts` is a custom `fetch` router that returns HTML strings.
- `apps/api/src/index.ts` owns `/api/auth/*`, device auth, billing/quota, devices, and Personal Memory APIs.
- `apps/web/src/index.ts` exports `createWebApp`, `createWebAuthClient`, `buildDeviceHandoffUrl`, and a `webAppBoundary` describing the current web boundary.
- `tests/web-account.test.ts` and `tests/web-download.test.ts` define the current web behavior to preserve.
- Root scripts use Bun, with `web:dev` currently running `bun --hot apps/web/src/dev.ts`.

## Recommended Approach

Use TanStack Start with Vite inside `apps/web`, keeping `apps/api` as the auth/API owner. Do not move Better Auth into the web app. The Start app should call the existing API endpoints from server-side route loaders/actions and preserve cookie forwarding to Better Auth.

## Implementation Steps

1. Add TanStack Start dependencies to `apps/web/package.json`.

- Runtime: `@tanstack/react-start`, `@tanstack/react-router`, `react`, `react-dom`.
- Dev: `vite`, `@vitejs/plugin-react`, `typescript`, `@types/react`, `@types/react-dom`, `@types/node` if needed locally by Start/Vite.
- Keep existing `@tabb/billing`, `@tabb/contracts`, and `better-auth` dependencies.

2. Add TanStack Start configuration.

- Create `apps/web/vite.config.ts` with `tanstackStart()` before `viteReact()` and `server.port = 3000`.
- Add or adjust `apps/web/tsconfig.json` only if app-local config is needed; otherwise rely on root `tsconfig.json` if Start accepts it.
- Update `apps/web/package.json` scripts: `dev: vite dev`, `build: vite build`, and a production start script if Start output requires one.
- Update root `package.json` `web:dev` to run the web workspace Start dev script with Bun.
- Update `scripts/dev.ts` if it assumes the old `apps/web/src/dev.ts` entrypoint.

3. Introduce the Start app skeleton.

- Create `apps/web/src/router.tsx` using `createRouter({ routeTree, scrollRestoration: true })`.
- Create `apps/web/src/routes/__root.tsx` with document shell, `HeadContent`, `Scripts`, shared layout, and global CSS import.
- Let `apps/web/src/routeTree.gen.ts` be generated by TanStack Start and keep it in source if the toolchain expects generated route trees to be committed.
- Add `apps/web/src/styles.css` for the web visual system.

4. Extract API client helpers for server-side route use.

- Add `apps/web/src/lib/api.ts` with `apiBaseUrl`, `apiRequest`, cookie forwarding, response parsing helpers, and redirect helpers.
- Use `TABB_API_BASE_URL` with default `http://localhost:8787`.
- Parse API responses with existing schemas from `@tabb/contracts`.
- Keep `buildDeviceHandoffUrl` exported from `apps/web/src/index.ts` for desktop tests/integration.

5. Implement auth routes.

- `/login`: render sign-in form, preserve `device_id` and `callback` query params, submit through a Start server action/server function to `/api/auth/sign-in/email`, set Better Auth cookies on success, and redirect to `/dashboard` or authorize device handoff.
- `/signup`: render account creation form, call `/api/auth/sign-up/email`, set session cookies or sign in immediately after signup, then redirect to `/dashboard`.
- `/logout`: POST-only sign-out action that calls `/api/auth/sign-out`, clears/forwards cookies as supported by the API response, and redirects to `/`.
- Protected routes should call `/api/auth/get-session` from server loaders/actions and redirect to `/login` when unauthenticated.

6. Implement public routes.

- `/`: landing page for Tabb with strong product positioning, download CTA, pricing CTA, permission/privacy trust notes, and responsive layout.
- `/pricing`: render `planQuotas` with checkout links for paid plans.
- `/download`: page linking to `/download/tabb.dmg`.
- `/download/tabb.dmg`: server route/handler redirecting to `TABB_MAC_DOWNLOAD_URL` or default artifact URL.
- `/download/latest.json`: server route/handler returning `{ version, url, notes }` from env/defaults.

7. Implement dashboard/account routes.

- Prefer `/dashboard` as the primary authenticated page and keep `/account` as a redirect or alias for compatibility.
- Dashboard loader fetches session, billing quota, devices, and memories in parallel using the request cookie.
- Dashboard overview shows current plan, monthly usage, reset date, upgrade warning, and billing portal link.
- Account configuration panel shows user identity and safe account settings placeholders that do not imply unsupported API capabilities.
- Usage page/section shows usage progress and plan limits from `BillingQuotaResponseSchema`.
- Permissions/devices page/section lists linked native devices, status, platform/version, and revoke action.
- Memory page/section lists Personal Memory entries and supports delete action.
- Keep POST actions equivalent to existing behavior:
  - `/account/devices/$deviceId/revoke` posts to `/api/auth/device/revoke` and redirects with `tab=devices`.
  - `/account/memory/$memoryId/delete` deletes `/api/account/memory/$memoryId` and redirects with `tab=memories`.

8. Preserve billing redirects.

- `/billing/checkout?plan=pro|max`: require session, call `/api/billing/checkout`, redirect to returned Polar URL.
- `/billing/portal`: require session, call `/api/billing/portal`, redirect to returned Polar URL.
- Keep error pages for failed checkout/portal API responses.

9. Update exports and remove old runtime entrypoints.

- Update `apps/web/src/index.ts` to export shared web boundary metadata, auth client helper, `buildDeviceHandoffUrl`, and any Start-compatible request handler if tests need direct invocation.
- Remove or stop using `apps/web/src/server.ts` and `apps/web/src/dev.ts` after tests are migrated.
- Keep minimal compatibility wrappers only if necessary for tests during the migration; do not preserve the old string-renderer long term.

10. Update tests.

- Migrate `tests/web-account.test.ts` and `tests/web-download.test.ts` away from the old `createWebApp().fetch` contract.
- Prefer testing Start request handlers directly if TanStack Start exposes an ergonomic handler in this setup; otherwise split into:
  - API interaction/unit tests for server helpers/actions.
  - Route rendering tests for key HTML content.
  - End-to-end smoke test against the Start dev/build server if practical in Bun.
- Preserve the existing assertions for marketing copy, pricing numbers, login fields, billing redirects, memory deletion, device revocation, download redirect, and latest JSON.

## Design Notes

- Keep auth source of truth in `apps/api`, matching existing ADRs and avoiding duplicated auth/session state.
- Prefer server-side loaders/actions for authenticated dashboard data so cookies stay server-side and API response parsing remains centralized.
- Use plain global CSS first instead of introducing Tailwind/shadcn; this is the smallest correct migration and keeps dependency scope controlled.
- Preserve existing product routes and desktop handoff URL semantics so the Electron app continues to open `/login?device_id=...&callback=...`.
- Do not invent unsupported account settings APIs. Show configurable sections only when they map to existing endpoints, or present placeholders as read-only until backing APIs exist.

Manual checks in browser:

- Visit `/`, `/pricing`, `/download`, and `/download/latest.json`.
- Create an account at `/signup`, sign out, and sign back in at `/login`.
- Confirm unauthenticated `/dashboard` redirects to `/login`.
- Confirm `/account` redirects or aliases to the dashboard.
- Confirm dashboard shows usage, plan, billing portal, devices, and memory.
- Revoke a linked device and delete a memory entry using test data or seeded API state.
- Confirm `/billing/checkout?plan=pro` and `/billing/portal` redirect through the API.
- Confirm desktop handoff login still redirects to the callback URL with an exchange code.
