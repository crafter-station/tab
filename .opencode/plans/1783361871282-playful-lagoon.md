# Plan: Show API Next Tokens In Debug Overlay

## Goal

Connect the captured typing context to the existing suggestion API and display the returned next-token suggestion in the overlay. Keep this first pass focused on the current typed text only: no memories, no broader context work, no new API surface unless the existing endpoint proves insufficient.

## Current State

- Native macOS typing capture emits text events from `apps/desktop/native/macos-input-tap.swift`.
- `apps/desktop/src/main.ts` receives text in `handleTextInput()`, appends it to `typingContextBuffer`, shows the debug overlay, and notifies `suggestionLoop`.
- `apps/desktop/src/suggestion-loop.ts` already debounces typed context, suppresses empty/paused/secure/private context, redacts secrets, calls `requestSuggestion(context)`, and shows returned suggestions.
- `apps/desktop/src/suggestion-client.ts` already posts to `POST /suggestions` and returns the first API suggestion.
- `apps/desktop/src/index.html` already renders either normal suggestions or the debug typing context, but debug mode hides normal suggestion state behind the raw context UI.
- `apps/api/src/index.ts` already implements `/suggestions` and the real model-backed generator.

## Recommended Approach

Reuse the existing `/suggestions` route and desktop suggestion loop, then make debug overlay rendering explicit about API state.

1. Disable memories for this first pass.
   - Update `apps/desktop/src/suggestion-client.ts` so the suggestion request can be configured with `memoryEnabled`.
   - Default to the current behavior if needed, but instantiate the desktop client in `apps/desktop/src/main.ts` with `memoryEnabled: false` for now.
   - Keep the payload as the current typing context plus active application metadata required by the existing schema.

2. Add lightweight suggestion-loop status callbacks.
   - Extend `apps/desktop/src/suggestion-loop.ts` dependencies with optional callbacks such as `onRequestStarted(context)` and `onRequestFinished(result)` or a single `onStatusChanged(status)`.
   - Fire status after debounce when a request is about to be made, when no suggestion is returned, and when a suggestion is shown.
   - Preserve current production behavior when callbacks are omitted.
   - Avoid changing the suppression/redaction behavior; secret-like text must still never reach the API.

3. Send debug API state to the overlay.
   - In `apps/desktop/src/main.ts`, track a small debug API state: `idle`, `loading`, `empty`, `error` if available, and `suggestion`.
   - Include this state in `sendDebugContext()` on the existing `debug-context` IPC payload instead of adding a separate IPC channel.
   - When the API returns a suggestion, continue using `showOverlay(suggestion)` so the real returned text is visible and can be accepted.
   - When the API returns no suggestion, keep the debug overlay visible and show a clear “No suggestion yet” style message rather than silently leaving only the raw context.

4. Update overlay UI copy/layout.
   - Modify `apps/desktop/src/index.html` debug rendering to show two sections: captured input and API response/status.
   - Keep the existing normal suggestion rendering for accept/click behavior.
   - Increase debug overlay height only if necessary in `apps/desktop/src/main.ts` so both input and response fit.

5. Keep API changes minimal.
   - Do not add a new endpoint for this pass.
   - Do not change `apps/api/src/index.ts` unless verification shows `/suggestions` still consults memories when `memoryEnabled: false`; if so, update the route to skip memory selection when the request disables memories.

## Files To Modify

- `apps/desktop/src/suggestion-client.ts`
- `apps/desktop/src/suggestion-loop.ts`
- `apps/desktop/src/main.ts`
- `apps/desktop/src/preload.ts` only if the debug payload type needs to include the new API state fields
- `apps/desktop/src/index.html`
- `apps/api/src/index.ts` only if `memoryEnabled: false` is not already honored server-side
- `tests/desktop-api-client.test.ts`
- `tests/desktop-native-loop.test.ts`
- `tests/api-suggestion.test.ts` only if API memory-skip behavior changes

## Verification

1. Run targeted tests:
   - `bun test tests/desktop-api-client.test.ts`
   - `bun test tests/desktop-native-loop.test.ts`
   - `bun test tests/api-suggestion.test.ts` if API behavior changes

2. Run the desktop app in debug typing mode against the local API:
   - Start the API with the existing project command.
   - Start the desktop app with `TABB_DEBUG_TYPING_OVERLAY=1` and the existing desktop dev command.
   - Type into TextEdit or another normal app.
   - Confirm the overlay shows captured input, then a loading/API status, then the model suggestion.
   - Confirm clicking or pressing Option+Tab still accepts the shown suggestion.

3. Privacy regression checks:
   - Type an obvious secret such as `api_key=sk-...` and confirm no API request is made and the buffer clears.
   - Confirm secure input, pause, and password-manager contexts still suppress requests.

## Notes

- This plan intentionally avoids memories and broader context by setting `memoryEnabled: false` and only sending the typing buffer.
- The existing API requires authentication; if the desktop is signed out, the expected first-pass behavior is no returned suggestion rather than a separate unauthenticated debug bypass.
