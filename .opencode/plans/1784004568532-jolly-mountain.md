# Restore standalone web-to-API development wiring

## Diagnosis

`bun run api:dev` correctly starts Wrangler on `localhost:8787`, but `bun run web:dev` loads `.dev.vars` where `PORT` is the API port and `WEB_PORT` is the web port. The current Vite config reads only `PORT`, and its Cloudflare runtime can retain the production `TAB_API_BASE_URL` from the web Wrangler config. This makes the separately started web app use the wrong port and/or backend, which explains why no requests reach the local API.

## Implementation

1. Update `apps/web/vite.config.ts` to make standalone and combined development deterministic.
   - Resolve the Vite server port from `WEB_PORT`, then `PORT`, then `3000`.
   - Validate the configured port and report an actionable error for invalid values.
   - Enable `server.strictPort` so an occupied web port fails visibly instead of shifting to an auth-incompatible origin.
   - Use Vite's command-aware config form to override the Cloudflare Worker's `TAB_API_BASE_URL` only during the dev server.
   - In dev, use a non-empty host `TAB_API_BASE_URL`, falling back to `http://localhost:8787`, while preserving all other Wrangler vars.
   - Do not apply the override to builds or preview, so `apps/web/wrangler.jsonc` remains the production source of `https://api.tab.cueva.io`.

2. Add a focused real-runtime regression test under `tests/` (for example, `tests/web-dev-config.test.ts`).
   - Start a local stub API and the actual web Vite dev configuration on distinct dynamic ports.
   - Supply both `PORT` as the occupied API port and `WEB_PORT` as the intended web port to reproduce the regression.
   - Supply the stub URL as `TAB_API_BASE_URL`, request a web route that performs session lookup, and assert that the stub API receives `/api/auth/get-session`.
   - Assert the web server listens on `WEB_PORT`, not `PORT`, and always terminate child/server resources.
   - Keep `tests/web-route-wiring.test.ts` unchanged unless a small shared test utility is genuinely needed; it uses `configFile: false` and virtual bindings, so it cannot cover this bug.

## Verification

1. Run the new focused dev-config regression test.
2. Run `bun test tests/web-route-wiring.test.ts` to preserve existing BFF behavior.
3. Run `bun run typecheck`.
4. Run `bun run --cwd apps/web build` to confirm production config still builds without the local API override.
5. Manually start `bun run api:dev` and `bun run web:dev`; verify Wrangler is on `8787`, Vite is on `3000`, and loading `http://localhost:3000/login` emits `/api/auth/get-session` in the local API logs.
6. Smoke-test `bun run dev` to confirm combined orchestration still uses the same `8787`/`3000` topology.
