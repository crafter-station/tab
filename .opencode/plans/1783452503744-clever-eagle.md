# Desktop Onboarding Flow Plan

## Goal

Build a first-run desktop onboarding flow shown only once after sign-in is required, then permissions are configured, then the user learns and practices accepting/rejecting autocomplete suggestions. If onboarding has already been completed or skipped, launch the main tab app instead. Keep the three desktop surfaces distinct: onboarding window, tab app/settings window, and tab overlay.

## Decisions

- Skip permanently marks onboarding complete and should not show again unless preferences are reset.
- Sign-in is required before continuing beyond the first onboarding step.
- Completion and skip should both close onboarding and open the tab app/settings window.
- The overlay remains the real suggestion overlay surface; onboarding should not merge with or replace it.
- Persist once-only state in the existing desktop preferences file at `app.getPath("userData")/preferences.json`.

## Implementation Steps

1. Extend the onboarding model
   - Keep `OnboardingPreferences.completed` as the durable gate.
   - Add lightweight onboarding step/copy constants in `onboarding.ts` for the expanded flow: sign-in, permissions, how it works, practice, done.
   - Keep `shouldShowOnboarding()` as `!completed`.
   - Keep `completeOnboarding()` for successful finish and skip, since both should suppress future onboarding.

2. Add a single startup surface decision
   - In `index.ts`, replace the current `if onboarding else if SHOW_SETTINGS_ON_START` branch with a small function such as `showInitialDesktopSurface()`.
   - If `onboardingManager.shouldShowOnboarding()` is true, show the onboarding window.
   - Otherwise show the settings/tab app window.
   - Keep creating the hidden overlay at bootstrap so autocomplete can still work independently once permissions/auth are ready.
   - Keep debug `SHOW_SETTINGS_ON_START` behavior only if still useful, but do not let it suppress the normal tab app open when onboarding is complete.

3. Make onboarding completion open the tab app
   - Update the `complete-onboarding` IPC handler in `index.ts` to mark onboarding complete, close onboarding, then call `settingsWindowManager.show()`.
   - Add a named renderer API like `skipOnboarding()` only if the UI needs semantic clarity; otherwise reuse `completeOnboarding()` for both Finish and Skip.
   - Ensure this behavior is also used when the user finishes the practice step.

4. Support required sign-in during onboarding
   - Reuse existing `window.tabb.signIn()`, `getInitialState()`, and `onStatusChanged()` APIs.
   - In `OnboardingSurface.tsx`, load initial status and subscribe to status changes, like `SettingsSurface` does.
   - Step 1 shows sign-in copy and a Sign In button.
   - Disable the Continue action until `status.auth === "signed_in"`.
   - After the `tabb://auth/callback` is handled in main, the existing status refresh should notify onboarding via `status-changed`; update `index.ts` so `statusService.onChange` sends status to both settings and onboarding windows if needed.
   - If onboarding window currently lacks a `sendStatus` method, add it to `createOnboardingWindowManager()` using the same pattern as `settingsWindowManager.sendStatus()`.

5. Expand permissions step
   - Preserve current Accessibility and Input Monitoring setup behavior.
   - Keep polling Accessibility every 1.5s after opening System Settings.
   - Keep Input Monitoring manual confirmation because macOS does not expose the same reliable check here.
   - Let the user continue after opening/confirming Input Monitoring, with clear relaunch guidance.

6. Add “how it works” education step
   - Explain that Tabb watches recent typing context in memory, requests a suggestion, then shows a floating overlay near the bottom of the active display.
   - Explain accept via Option+Tab or clicking the suggestion.
   - Explain reject/dismiss by ignoring it, pressing Escape if implemented already, or continuing typing so the suggestion becomes stale. If no explicit reject control exists, phrase this as dismissal rather than adding a real reject button to the production overlay.

7. Add practice step inside onboarding
   - Build a local mock practice area in `OnboardingSurface.tsx`, not a real API-backed suggestion flow.
   - Include a textarea/input with prompt copy such as “Write a short reply...” and a mock suggestion pill styled like the real overlay.
   - Provide three practice actions:
     - Approve: appends the mock completion to the practice text and records approval.
     - Reject: hides/replaces the mock suggestion and records rejection.
     - Try again: regenerates or restores a mock completion.
   - Gate Finish until the user has practiced at least approving and rejecting once, or provide a clear “Finish anyway” action if the UI would otherwise trap them.
   - This avoids triggering the real overlay or pasting into other apps during onboarding.

8. Add Skip affordance
   - Put a small `Skip setup` button in the onboarding footer/header on every step.
   - On click, call the same completion path: mark completed, close onboarding, open the tab app.
   - Use copy that makes the consequence clear: “You can configure this later in Settings.”

9. Update onboarding window sizing/styles
   - Increase `onboarding-window.ts` dimensions if needed for the multi-step content, while keeping the fixed welcome-window feel.
   - Update `onboarding.css` for 5-step progress, sign-in status card, tutorial panels, and practice suggestion controls.
   - Reuse `Button`, `PermissionCard`, `SectionCard`, `StatusBadge`, and existing dark/glass styling.

10. Update preference validation/tests
   - If the preference shape remains `{ onboarding: { completed: boolean } }`, keep validation simple and only update tests for skip/complete semantics.
   - If adding extra onboarding fields, make `isDesktopPreferences()` tolerate older preference files by merging parsed values onto defaults rather than rejecting older files.
   - Add/adjust tests for:
     - default onboarding shows once.
     - complete/skip marks complete.
     - completed onboarding suppresses onboarding.
     - preferences persist the completed flag.
     - any new copy still avoids “keylogger”/raw keystroke language and still excludes Screen Recording/Full Disk Access from requested permissions.

## Verification

1. Run `bun test tests/desktop-onboarding.test.ts tests/desktop-preferences.test.ts`.
2. Run `bun run typecheck`.
3. Run `bun run desktop:dev` with a clean or reset desktop preferences file.
4. Verify first launch opens onboarding, not the tab app/settings window.
5. Verify Sign In opens the browser handoff and onboarding cannot continue until status becomes `signed_in`.
6. Verify permissions step opens the correct macOS settings panes and Accessibility polling advances when granted.
7. Verify practice step lets the user approve and reject mock completions without invoking the real overlay paste path.
8. Verify Finish closes onboarding, persists `onboarding.completed: true`, and opens the tab app/settings window.
9. Reset preferences, relaunch, click Skip, then verify onboarding does not show again and the tab app opens.
10. Relaunch with completed preferences and verify onboarding is skipped, the tab app opens, and the hidden overlay still initializes normally.
