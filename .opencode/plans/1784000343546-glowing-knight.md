# TanStack Start And Shared UI Cutover

## Goal

Make TanStack Start the single web runtime so `@tab/ui` shadcn/Radix components are genuinely hydrated, remove the static/custom component fork, and replace hardcoded application controls and surfaces with shared design-system primitives. Preserve native links and forms without JavaScript; menus, sheets, dialogs, tabs, and toggles may require hydration.

## Architecture

- Replace the active `createWebApp` + `renderToStaticMarkup` request path with TanStack Start SSR and hydration. The existing Router-only route tree is not sufficient; use Start server functions, server route handlers, middleware, and the official Cloudflare/Vite integration.
- Keep `apps/api` as the sole auth, billing, device, activity, and Personal Memory authority. The web Worker remains a BFF that reads the incoming request, forwards cookies/origin to the API, validates responses with `@tab/contracts`, and relays all `Set-Cookie` headers.
- Use protected `createServerFn` calls for typed internal reads such as viewer/session and dashboard data. Route loaders call these functions; loaders must not directly import request-header APIs.
- Use route `server.handlers` for stable native form POSTs, raw JSON/download/export responses, and redirects requiring exact HTTP semantics.
- Use dashboard `beforeLoad` only for signed-out UX redirects. Every protected function and handler must independently authenticate.
- Remove the old renderer after the Start routes reach behavior parity. Do not retain static and hydrated component implementations as a compatibility architecture.

## Slice 1: Shared UI Foundations

### Add missing primitives

- Add shadcn-style `Dialog` in `packages/ui/src/components/ui/dialog.tsx`, using the existing Radix Dialog dependency, Phosphor icons, and Tab radius/shadow/motion tokens.
- Add shadcn-style `AlertDialog` in `packages/ui/src/components/ui/alert-dialog.tsx` and add its Radix dependency.
- Export both from `packages/ui/src/index.ts`; update `packages/ui/package.json` and `bun.lock`.
- Do not add `Popover`, `Accordion`, or `Collapsible` without a concrete remaining use. Semantic content disclosures can remain native `<details>`.

### Deepen existing app patterns

- Update `packages/ui/src/components/app/patterns.tsx`, `section-card.tsx`, and `status-badge.tsx` only as needed to support the existing dashboard states without dashboard-local status dots or card recipes.
- Keep domain-specific compositions such as quota progress and pricing plans in `apps/web`; shared UI should remain domain-neutral.
- Extend `tests/ui-patterns.test.tsx` for Dialog/AlertDialog SSR structure, accessible title/description requirements, and any pattern API changes.

## Slice 2: Complete The TanStack Start Runtime

### Runtime and deployment configuration

- Update `apps/web/vite.config.ts`, `apps/web/package.json`, `apps/web/wrangler.jsonc`, root `package.json`, `scripts/dev.ts`, and `bun.lock` to use Vite/Start for development, build, preview, and deployment.
- Add the official Cloudflare Vite plugin and follow current TanStack Start/Cloudflare plugin guidance, keeping `tanstackStart()` before the React plugin. Preserve the custom domain, Worker compatibility flags, vars, and observability.
- Make Start's server entry the Worker entry instead of `dist/server/server.js`; verify actual build output and Wrangler packaging rather than hardcoding an assumed assets directory.
- Let Vite process `apps/web/src/styles.css`, font imports, and `public/` assets. Remove the manual generated-CSS/font-copy pipeline only after build and preview prove parity.
- Resolve the Start TypeScript requirement around `verbatimModuleSyntax` with the narrowest web-specific tsconfig change that prevents server modules leaking into client output.

### Server-only BFF modules

- Split `apps/web/src/lib/api.ts` into clear client-safe and server-only boundaries, introducing focused modules under `apps/web/src/lib/` for runtime env, API forwarding, session/auth, auth middleware, dashboard server functions, mutation handlers, and validated search/form inputs.
- Read request/cookie data through `@tanstack/react-start/server` only inside server functions, middleware, or handlers.
- Preserve injectable `fetch` and base-URL seams for in-memory API tests without recreating a second web router.
- Add a shared session-user schema to `@tab/contracts` if no existing schema validates the Better Auth session shape.
- Mark viewer/dashboard responses private and non-cacheable; never serialize cookies, device tokens, upstream headers, or secrets into loader data.

### Root and public routes

- Convert route modules to file-based `createFileRoute` definitions and let Vite regenerate `apps/web/src/routeTree.gen.ts`; never edit it manually.
- Update `apps/web/src/routes/__root.tsx` to own the HTML document, global CSS, metadata, `HeadContent`, `Scripts`, theme initialization, error/not-found UI, optional viewer data, and the single hydrated site shell.
- Complete `/`, `/pricing`, `/brand`, `/about`, `/contact`, `/privacy`, `/terms`, `/download`, and `/components`, preserving current metadata, signed-in header behavior, and real 404 status.

### Auth and stable POST routes

- Implement GET UI plus POST `server.handlers` for login, signup, forgot password, reset password, and logout in their existing route files.
- Preserve desktop handoff fields and callback behavior, safe `next` validation, email-verification flow, origin forwarding, and every API `Set-Cookie` header.
- Keep passwords out of redirect URLs. Use POST/Redirect/GET and `303` for successful mutations; represent recoverable errors with validated, non-sensitive route state.
- Redirect authenticated ordinary login/signup visits to `/dashboard` while preserving authenticated device handoff behavior.

### Dashboard data and mutations

- Make `apps/web/src/routes/dashboard.tsx` the authenticated layout boundary. Its `beforeLoad` redirects signed-out users and its parent loader calls one protected server function that loads billing, devices, memories, and local activity in parallel.
- Add a dashboard index route and have child routes consume parent loader data rather than pathname inference or a manually maintained data context.
- Add raw server-handler route files for device revoke, memory create/edit/delete/bulk-delete/export, preserving current form names, confirmation values, authorization checks, response headers, and redirect destinations.
- Implement billing checkout/portal, DMG redirect, latest-version JSON, and account redirects with the appropriate loader/server-handler primitive. Preserve monthly-only checkout normalization and all current paid-plan/error behavior.
- Search for real callers of legacy `/account/...` mutation URLs. Add thin aliases only where a concrete external or persisted compatibility need exists.

### Remove the obsolete runtime

- Delete `apps/web/src/server.ts`, `apps/web/src/dev.ts`, and `apps/web/src/render-page.tsx` once request-level parity is green.
- Remove `createWebApp`, `WebApp`, and server-only exports from `apps/web/src/index.ts`; retain only genuinely public/client-safe helpers such as desktop handoff URL construction.
- Remove generated CSS artifacts and old instructions only after Vite owns those assets; update `AGENTS.md`, README/contributor commands, and related contract tests to describe the real Start runtime.

## Slice 3: Eliminate Custom Menus And Dashboard Forks

### Site shell menus

- In `apps/web/src/components/brand-menu.tsx`, keep one hydrated `BrandMenu`, delete `StaticBrandMenu`, and use `DropdownMenuRadioGroup`/`DropdownMenuRadioItem` for theme selection.
- In `apps/web/src/components/site-shell.tsx`, replace desktop “More” `<details>` with `DropdownMenu` and mobile navigation `<details>` with the shared `Sheet`.
- Add a single hydrated user menu composed from `DropdownMenu`, native links, and a native logout form. Remove static theme/user menu implementations with `render-page.tsx`.
- Preserve `href`, `method`, and `action` semantics so navigation and forms remain native even though opening a menu requires hydration.

### Dashboard structure

- Split `apps/web/src/components/pages/dashboard.tsx` into focused modules under `apps/web/src/components/dashboard/` for layout/navigation, overview, account, usage, devices, memories, and row actions.
- Keep only the shared `Sidebar`, `SidebarProvider`, `SidebarTrigger`, and related primitives. Remove `DashboardStaticBrandMenu`, `DashboardStaticSidebar`, `DashboardStaticLayout`, static sidebar scripts/CSS, pathname section inference, and duplicate layout branches.
- Replace `DashboardMetric` and ad hoc status/surface recipes with `SummaryMetric`, `SectionCard`, `StatusRow`, `StatusBadge`, `Alert`, and existing table primitives.
- Keep `QuotaProgressPanel` domain-specific.

### Correct interaction semantics

- Device actions: `DropdownMenu` command opens `AlertDialog` containing the native revoke form and confirmation input.
- Memory actions: `DropdownMenu` commands open a `Dialog` for the native edit form and an `AlertDialog` for the native delete form. Remove nested `<details>` action panels.
- Bulk deletion: use `AlertDialog` and shared `Checkbox`, preserving actual native `FormData` serialization for `memoryId` and the confirmation value. Verify Radix Checkbox behavior in Chromium; use synchronized hidden native inputs only if necessary.
- Keep “Add a memory” as a semantic native `<details>` so the form remains available without JavaScript.
- Verify outside click, Escape, keyboard navigation, and focus return for every menu/dialog path.

## Slice 4: Remaining Hardcoded UI Cleanup

### Pricing and cards

- Add app-local `PricingPlanCard`/grid components under `apps/web/src/components/pricing/`, composed from shared `Card`, `Badge`, and `Button` primitives.
- Use that composition on both home and pricing pages, with explicit summary/full and featured treatments instead of duplicated markup.
- Replace native buttons styled with `buttonVariants` by `Button`; wrap CTA anchors with `Button asChild` while preserving native links and GET checkout forms.

### Marketing interactions

- Move the autocomplete demo, replay behavior, motion state, and tab keyboard behavior from `apps/web/public/marketing-demo.js` into focused React components under `apps/web/src/components/marketing/`.
- Replace hand-built tabs with shared `Tabs`, motion controls with `Toggle`, and replay controls with a reusable `Button` composition.
- Preserve Option+Tab behavior, reduced-motion handling, accessible state/announcements, and existing animation hooks where still useful.
- Delete `marketing-demo.js`, its script tag, and obsolete script-specific tests after browser coverage replaces them.

### Intentional bespoke UI

- Retain native `<details>` for FAQ, billing/trial explanation, and Add Memory because they are semantic disclosures, not menus.
- Do not mechanically convert product illustrations, workflow maps, brand previews, legal sections, contact links, auth handoff content, or quota progress into generic cards. Use shared controls/status/surfaces within them where appropriate while preserving their semantics and visual purpose.
- Remove or narrow unused barrels such as `apps/web/src/components/web-pages.tsx` after checking all imports.

## Testing Strategy

### Preserve server behavior

- Refactor `tests/web-account.test.ts`, `web-download.test.ts`, `web-brand.test.ts`, and `web-font-assets.test.ts` away from `createWebApp`.
- Keep the existing in-memory Hono API fixture and test server helpers/actions directly for auth, cookie/origin forwarding, billing, device, memory, and download behavior.
- Add request-level coverage through the built Start handler or a production-like test server for statuses, redirects, cookies, SSR metadata/content, 404s, raw JSON, and downloads.
- Preserve all existing behavioral contracts, updating successful POST expectations to `303` where PRG is introduced.

### Add real browser coverage

- Add Playwright configuration and a focused web interaction suite backed by the in-memory API fixture.
- Cover: logo/user/navigation dropdown outside-click dismissal; Escape dismissal; focus return; theme radio persistence; mobile Sheet behavior; device and memory menu-to-dialog flows; bulk checkbox form submission; Tabs keyboard behavior; Toggle/replay accessible state.
- Add a JavaScript-disabled project for public links, auth/password forms, checkout, Add Memory, and direct mutation form semantics. Do not require dropdowns, sheets, dialogs, tabs, or toggles to work without JavaScript.
- Prefer behavior assertions over source-string assertions; retain only architecture boundary tests that prevent reintroducing custom menus or the static renderer.

## Verification

Run incrementally after each slice, then complete CI order from the repository root:

```sh
bun install --frozen-lockfile
bun run typecheck
bun run worker:types:check
bun run lint
bun run test
bun run --cwd apps/web build
```

Also run the new Playwright web suite and the installed Wrangler version's supported dry-run/package validation. Manually smoke-test Vite development and a production-like preview for:

- Public SSR source, metadata, CSS, fonts, and brand assets.
- Signed-in and signed-out headers; auth, logout, verification, and desktop handoff cookies.
- Safe dashboard and checkout redirects.
- All dashboard sections and native mutations.
- JSON/export/download response headers.
- Desktop/mobile menus, dialogs, outside click, Escape, focus return, and theme persistence.
- No accidental local calls to the production API.

## Delivery Order And Risk Controls

1. Shared UI primitives and static component tests.
2. Server helper extraction plus characterization tests while the old runtime still describes behavior.
3. Atomic Start runtime/config/route cutover, request tests, and deletion of the old renderer.
4. Menu/dashboard migration with browser interaction tests.
5. Pricing and marketing primitive cleanup with responsive and reduced-motion review.

Treat these as reviewable slices of one approved effort. Do not merge a state where production still targets the custom renderer after routes have diverged. Deployment gates are Start build success, Wrangler packaging, cookie forwarding across differing web/API hosts, private caching for user data, deterministic route-tree generation, and Chromium verification of form-associated Radix controls.
