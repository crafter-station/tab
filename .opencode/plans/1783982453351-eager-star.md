# Polar Production And Monthly Max Plan

## Goal

Replace the active paid catalog with monthly Pro (`$10`, 300 Deep Completes) and monthly Max (`$20`, 1,000 Deep Completes), then provision those resources in Polar production and configure the production Cloudflare Worker. Free and the 30-day Pro trial remain. Max otherwise has the same capabilities and three-device limit as Pro. There are no annual customers to preserve.

## Preconditions

- Confirm ignored `dev.vars.prod` contains `POLAR_ACCESS_TOKEN`, `POLAR_SERVER=production`, and the `POLAR_ORGANIZATION_ID` the user will add. Never print or commit their values.
- Verify the Polar organization is active and has API, checkout-payment, and subscription-renewal capabilities before creating resources.
- Verify Wrangler is authenticated to the Cloudflare account that owns `tab-api`.

## Implementation

1. Update the canonical plan model and contracts.
   - In `packages/billing/src/index.ts`, define Free, Pro, and Max; remove active annual pricing; add Max at `$20/month` with 1,000 Deep Completes and the same remaining capabilities as Pro.
   - In `packages/contracts/src/index.ts`, add Max through the shared plan union and make active checkout monthly-only. Remove annual from runtime status because no production annual customers exist.
   - Update capability-driven messages in API and desktop code so paid features do not say they require Pro when Max also qualifies.

2. Generalize billing and checkout from interval-based Pro products to plan-based monthly products.
   - In `apps/api/src/billing.ts`, support Pro and Max as distinct paid entitlements, map each Polar product ID explicitly, preserve Max in D1/telemetry/webhook data, remove all legacy `max -> pro` normalization, and remove annual product requirements.
   - Refactor checkout configuration to use Pro and Max monthly product IDs. Allow each valid paid plan to select its own product while still failing clearly on partial production configuration.
   - In `apps/api/src/routes/billing.ts`, accept only Pro or Max monthly checkout. Same-plan requests are no-ops; active Pro-to-Max or Max-to-Pro changes continue through the Polar Customer Portal per ADR-0041 rather than creating a second subscription.
   - Keep the existing D1 schema: its text columns already support Max, so no migration should be generated.

3. Align all product surfaces and sources of truth.
   - Update pricing, home, dashboard, terms, route metadata, and checkout forwarding under `apps/web/src/` to show Free, Pro, and Max; remove annual pricing/selectors; and use plan-neutral upgrade/manage copy.
   - Update desktop settings copy and status fixtures so Max renders correctly.
   - Update `docs/PRD.md` and `docs/product-strategy.md` to the approved catalog.
   - Add a new ADR under `docs/adr/` that supersedes ADR-0043's Free/Pro-only and annual-price decisions while retaining its local-first metering policy. Preserve historical ADRs unchanged.
   - Update `.dev.vars.example`, `apps/api/src/env.ts`, and `scripts/env.ts` with unambiguous current keys: `POLAR_PRODUCT_ID_PRO_MONTHLY`, `POLAR_CREDITS_BENEFIT_ID_PRO_MONTHLY`, `POLAR_PRODUCT_ID_MAX_MONTHLY`, and `POLAR_CREDITS_BENEFIT_ID_MAX_MONTHLY`; remove active annual and retired Max aliases where no longer consumed.
   - Regenerate `worker-configuration.d.ts`; do not edit generated route trees or migrations.

4. Make Polar provisioning safe and idempotent.
   - Refactor `scripts/polar-migrate-local-first.ts` to accept an explicit environment-file path and atomically rewrite that same file rather than hardcoding `.dev.vars`. Refuse a production/file mismatch and never log secret values.
   - Provision or discover by validated metadata: one `deep_complete.used` meter, separate no-rollover credit benefits for Pro (300) and Max (1,000), and monthly fixed-price products at `$10` and `$20` with the matching benefit attached.
   - Refactor `scripts/polar-setup.ts` to discover/update the existing raw webhook endpoint instead of creating duplicates. Use `https://api.tab.cueva.io/api/billing/webhook` and the subscription events already handled by the API.
   - Add a read-only verification mode that checks organization identity/capabilities, meter shape, benefit quantities, product prices/intervals/attachments, and webhook URL/events without creating customers or usage events.
   - Update `scripts/polar-verify-usage.ts`, `scripts/polar-diagnose-customer.ts`, and root package commands for the two current plans and explicit production env-file usage.

5. Test the model before touching production.
   - Update billing-domain and PRD contract tests for Free/Pro/Max and monthly-only sales.
   - Update API tests for Max persistence, quota enforcement at 1,000, explicit product/webhook mapping, monthly-only checkout, same-plan behavior, and portal-based Pro/Max plan changes.
   - Update web and desktop tests for three pricing choices, no annual option, Max checkout/status, and neutral paid-plan language.
   - Add mocked provisioning tests covering idempotent discovery, exact prices/credits, benefit attachment, webhook reuse, env-file targeting, and refusal to mix sandbox and production.
   - Run `bun run typecheck`, `bun run worker:types:check`, `bun run lint`, and `bun run test`.

6. Provision Polar production only after the code and tests pass.
   - Run the env-aware migration against `dev.vars.prod`; capture generated meter, benefit, and product IDs back into that ignored file.
   - Create or reconcile the production webhook and store its returned signing secret in `dev.vars.prod` without printing it.
   - Run the new read-only production verifier. Do not run the existing synthetic-customer usage verifier against production.

7. Configure and deploy Cloudflare.
   - In `wrangler.jsonc`, set `POLAR_SERVER=production` and add the non-secret meter and Pro/Max product IDs as Worker vars. Keep the existing production checkout success URL and `POLAR_SEND_ORGANIZATION_ID=false`, as Polar documents that an Organization Access Token is already organization-scoped.
   - Upload only `POLAR_ACCESS_TOKEN` and `POLAR_WEBHOOK_SECRET` as Wrangler secrets, piping them from `dev.vars.prod` without placing values in command arguments or logs. Confirm with `wrangler secret list`.
   - Regenerate/check Worker bindings, deploy `tab-api`, and verify the custom API domain. No D1 migration is expected.

## End-To-End Verification

- Confirm the deployed billing status exposes Free, Pro, and Max capabilities correctly.
- Confirm unsigned/invalid webhook requests return `400`.
- With a production test account, create Pro checkout and verify `$10/month`, dashboard redirect, webhook-driven Pro entitlement, and customer portal access.
- Change Pro to Max through the portal and verify the webhook changes the cached plan to Max with a 1,000 monthly limit and no second subscription.
- Confirm a successful Deep Complete records one usage event and decrements Tab's D1 allowance; failures and empty results do not count.
- Re-run the repository CI command sequence after all generated/configuration changes and report any checks that cannot be completed safely.

## Documentation Basis

- Polar Organization Access Tokens are organization-scoped: https://polar.sh/docs/integrate/authentication
- Polar credit benefits renew on the subscription cycle, which is why annual sales are removed rather than pretending yearly credits replenish monthly: https://polar.sh/docs/features/benefits/credits
- Polar does not enforce credit exhaustion; Tab remains authoritative for monthly limits: https://polar.sh/docs/features/usage-based-billing/credits
- Polar raw webhooks use Standard Webhooks signatures and a signing secret: https://polar.sh/docs/integrate/webhooks/endpoints
