# Desktop Owned Temporary Memory Extraction Window

Tab keeps the recent Memory Extraction Window on the desktop rather than durably storing raw recent typing context in the backend. The desktop sends the bounded window to `POST /api/memory/extract` with device auth rather than enqueueing extraction from the hot suggestion path. The window is limited to at most 30 minutes and at most 8 KB of redacted eligible text entries, whichever is reached first; each buffered entry is capped at 1 KB. The desktop should send a non-full window after 60 seconds of idle time when it has at least 500 characters or 5 entries, or when the 30-minute limit is reached.

The API is called on an idle/debounced background cadence with bounded retry and an idempotency key. The desktop retries failed batches up to 3 times with exponential backoff, then drops them so it does not become a durable typing archive. Backend idempotency stores only temporary batch metadata such as user ID, batch ID hash, result counts, and a 24-hour expiry, not extraction-window content. For MVP, the API processes extraction synchronously and returns final operation counts after applying changes. On success, the desktop clears the processed buffer; on extraction failure, it keeps the batch only for the retry budget. No-op business results are not retained. A brief durable coordination row may exist solely to hand the result to concurrent cross-worker callers, and is physically pruned after that handoff window expires.

When memory is disabled, suggestions do not retrieve Personal Memory and the desktop does not buffer extraction-window entries; existing memories remain visible and deletable. Buffered entries contain already-redacted eligible user-authored text with minimal metadata such as timestamp, active application bundle ID, context source, and redaction summary; eligible sources include typed text and terminal user input, while pasted text and terminal output remain ineligible. The buffer does not include suggestion text, accepted suggestion text, window titles, document names, URLs, or ambient application contents.

The backend retrieves relevant existing Personal Memory records through Cloudflare Vectorize and fetches canonical records from D1, while raw extraction-window text is deleted after processing or TTL expiry so durable storage remains limited to the resulting Personal Memory records and their retrieval index entries. Extraction prompts include the recent window and top candidate memories retrieved from Vectorize using the whole window text; they instruct the LLM to store stable facts useful for future autocomplete, prefer no operation, avoid raw logs and speculative facts, never store secrets, payment data, or government identifiers, and update or delete only with direct evidence. Rejected operations are counted when they fail backend safety, ownership, cap, user-created-memory protection, or delete-reason checks.
