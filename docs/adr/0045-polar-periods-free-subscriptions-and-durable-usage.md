# Polar Periods Free Subscriptions And Durable Usage

Status: accepted. This ADR supersedes ADR-0043's account-authoritative UTC calendar month, paid-only Polar ownership, and exclusion of Local Accepted Words from Polar reporting. It extends ADR-0025 and ADR-0028 without changing Tab's local enforcement authority.

Free, Pro, and Max are monthly recurring Polar products. After a verified signup, Tab immediately grants local Free access and asynchronously provisions one Polar customer, identified by the Tab user ID as `external_id`, and one free monthly subscription. Polar unavailability never blocks signup. While provisioning is pending, D1 uses one stable provisional period anchored to account initialization so Free access remains usable; when Polar returns exact boundaries, the same transaction installs them and moves provisional Deep Complete usage into the authoritative period. The provisional boundary is never treated as a Polar lifecycle fact. D1 records pending, provisioning, retrying, ready, or failed state so scheduled reconciliation can safely repair existing and partially provisioned accounts.

For every plan and trial, Polar's exact `current_period_start` and `current_period_end` define the Deep Complete allowance period. The D1 period key combines the subscription ID and period start; Tab never models a Polar month as 30 days or a UTC calendar month. D1 remains the hot-path quota authority and reconciles expired or missed lifecycle state asynchronously rather than querying Polar for each Suggestion.

The Free Local Accepted Word allowance remains Mac-local, resets at local midnight, and uses the durable offline Acceptance ledger. Ledger events are scoped to the authenticated Tab user so pending events cannot cross accounts after sign-out. Ambiguous events from the older installation-scoped ledger are not reassigned. Multiple offline Macs cannot coordinate this allowance in real time; server reconciliation is monotonic and this is an accepted local-first tradeoff. Polar receives `local_accepted_words.used` with the Acceptance ID, original Acceptance timestamp, word count, and local day for reporting only. Polar credit balances never authorize or block Acceptance.

Successful Deep Completes, successful Rewrites, and newly deduplicated Local Accepted Word uploads create D1 outbox records before the API operation completes. A scheduled Worker delivers these records to Polar with stable external event IDs, leases, exponential retry, and visible terminal failure state. Typed text and Suggestion text are never included.

Paid checkout receives the existing Free Polar subscription ID so payment upgrades that subscription instead of creating another active subscription. After that upgrade, customers make every Plan Change in the Polar Customer Portal. Tab retains the current cached product until Polar applies a scheduled change and projects the resulting `subscription.updated` event. A period-end paid cancellation keeps paid access through the cached end and then provisions a replacement Free subscription after revocation. Stale cancellation or revocation events cannot mutate a replacement subscription.
