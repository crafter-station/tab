# Unstructured Personal Memory For Learned User Facts

Tab stores learned user facts, including identity and contact facts such as name or email, as ordinary Personal Memory records rather than as a separate typed autofill profile or field-specific database objects. Personal Memory records do not carry category, sensitivity, extraction-source, or active-state fields; existence means visible and usable, while deletion removes both the D1 record and its Vectorize entry. Each record carries `createdBy`, distinguishing explicit user-created memories from system-created memories learned by Tab.

Memory maintenance is LLM-driven over a periodic extraction window rather than a single message or suggestion request, and may create, update, or hard delete unique system-created memory records. D1 keeps the canonical memory model intentionally generic and Cloudflare Vectorize indexes each durable memory's content by memory ID for semantic relevance retrieval. Vectorize entries include generic metadata such as `userId` and `createdBy`; D1 remains the source of truth and ownership is verified after fetching records. Memory content is capped at 500 characters, and the backend enforces generic ownership, safety, integrity, and a 500 memories per-user cap rather than field-specific uniqueness rules.

Suggestion-time retrieval queries Vectorize with the current redacted typing context, while active application metadata remains prompt context rather than embedded query text; retrieval filtering is limited to generic ownership and score threshold checks. The suggestion LLM receives the full content of a small number of top retrieved Personal Memory records. If Vectorize retrieval fails, suggestions continue without memory rather than failing autocomplete.

The LLM owns semantic deduplication using retrieved candidate memories, and each operation includes a reason. Extraction operations use only create/update/delete with content and reason rather than model-invented category, sensitivity, extraction-source, or authorship fields. The backend sets extracted memories to `createdBy: "system"` and explicit user actions create `createdBy: "user"` memories. User-created memories can be used for relevance and deduplication context, but background LLM maintenance must not create, update, or hard-delete them. If a user edits a system-created memory, it becomes `createdBy: "user"`.

At the cap, memory maintenance may update or hard-delete existing system-created records but cannot create new ones. Hard delete operations require a model-provided reason grounded in direct contradiction or invalidation from the recent extraction window, not mere absence of evidence; the backend enforces non-empty reasons, valid ownership, and user-created-memory protection while the prompt carries the semantic contradiction rule. User deletion is always hard delete for both user-created and system-created memories.

Memory writes update D1 and Vectorize together: create/update writes the canonical D1 record and upserts the Vectorize entry, then revalidates the owner-scoped canonical record so a concurrent delete cannot resurrect a vector and a concurrent update cannot leave stale vector content. Delete atomically removes the canonical D1 record and records an owner-scoped Vectorize deletion tombstone; Vectorize cleanup and tombstone acknowledgement are idempotent and retryable on later memory requests. Canonical deletion determines extraction counts, so a cleanup failure does not reject or double-count an applied deletion. Extraction batches are atomically claimed in D1 with bounded leases, and concurrent requests wait for and reuse the owner-scoped completed result. A repair or reindex script can be added later. For MVP, users see the current Personal Memory list ordered by newest updated first rather than a full change history, and telemetry may record aggregate operation counts, latency, model ID, and failure codes without storing memory or deleted memory content.
